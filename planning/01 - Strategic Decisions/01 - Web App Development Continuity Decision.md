# Web App Development Continuity Decision

---

## 1. Decision Statement

Neuroartan will prioritize the web app as the primary development surface for the next thirty days.

The immediate objective is to build a modular AI Development Cockpit inside the website layer so development can continue even if external premium tooling access becomes limited.

This decision prioritizes a cloud-capable, repository-aware, prompt-driven web workflow over a software-first local runtime replacement.

---

## 2. Strategic Context

The current website platform has reached a usable direction, with an increasingly mature homepage shell, platform menu, product system, token architecture, and modular design language.

The current software layer is not mature enough to replace ChatGPT or Codex within one month. The software layer remains important, but it is not the safest immediate dependency for maintaining development velocity.

The immediate risk is development continuity.

The selected path must preserve the ability to generate scans, analyze files, create patch plans, prepare Codex prompts, manage GitHub workflows, and continue surgical development under limited AI-tooling availability.

---

## 3. Decision Rationale

The website layer is the most mature active surface.

The software layer is currently minimal and should not become the primary dependency for this month’s continuity mission.

A web-based AI Development Cockpit can be developed incrementally, modularly, and visibly.

A cloud/provider abstraction can allow free-tier or low-cost model experimentation without blocking the interface architecture.

A local model/runtime path should remain available as a future fallback, but it should not consume the main execution bandwidth during the current month.

---

## 4. Selected Operating Allocation

The operating allocation for the next thirty days is:

| Track | Allocation | Purpose |
|---|---:|---|
| Web App / Website Platform | 70% | Build the AI Development Cockpit, profile/model creation surfaces, prompt composer, and workflow UI. |
| API / GitHub / Terminal Bridge | 20% | Create practical development workflows for scan generation, file targeting, patch planning, and commit messages. |
| Local Runtime / GGUF Fallback | 10% | Maintain local model work as a fallback and future sovereignty layer, without making it the main dependency. |

---

## 5. Mission Outcome

By the end of the thirty-day cycle, the website layer should contain a working AI Development Cockpit with the following components:

- Prompt composer
- Terminal scan generator
- File path target selector
- Patch instruction generator
- GitHub commit message generator
- “Open this file next” workflow
- Model provider abstraction
- Cloud provider route
- Local provider route
- Manual/offline provider route
- Codex prompt queue
- Daily execution record system

---

## 6. Execution Doctrine

All development must remain modular.

No single overloaded file should become the cockpit.

The architecture must use separate HTML fragments, CSS files, JavaScript modules, data files, and planning documents wherever ownership boundaries are clear.

No workaround, overlay, or compensating patch may be accepted as a final implementation.

Every major implementation must follow:

1. Scan
2. Identify owner
3. Edit one owner
4. Verify behavior
5. Document decision or continuation when needed

---

## 7. Codex Usage Doctrine

Codex should be used through precise daily prompts, not broad unfocused instructions.

Each Codex prompt should define:

- Target folder
- Files to create or edit
- Architecture boundaries
- Required modularity
- Forbidden behavior
- Verification command
- Expected completion output

Codex should not be asked to redesign the whole system in one pass.

Codex should be used for structural creation and controlled implementation, while ChatGPT should remain the surgical review, correction, and strategic continuity layer during the available period.

---

## 8. Provider Strategy

The cockpit must support provider abstraction from the beginning.

The system should not be locked to one model provider.

Initial provider categories:

| Provider Mode | Role |
|---|---|
| Cloud | Online free-tier or low-cost model access for high-velocity reasoning and code assistance. |
| Local | Local GGUF or compatible runtime fallback for offline continuity and sovereignty. |
| Manual | Human-assisted prompt/result transfer when automation is unavailable. |

Provider integration must be modular and replaceable.

---

## 9. Software Layer Position

The software layer remains strategically important but secondary for this month.

The local runtime direction should continue only as a fallback and future product/runtime path.

The software layer should not become the critical path until the web app cockpit can support development continuity.

---

## 10. Website Layer Position

The website layer is the active execution surface.

The website should become the command interface for:

- Prompting
- Scanning
- Repository navigation
- File targeting
- Development planning
- Model routing
- GitHub workflow support
- Daily execution control

This does not turn the public website into an exposed internal tool. The cockpit must be developed in a controlled architecture with appropriate visibility boundaries.
