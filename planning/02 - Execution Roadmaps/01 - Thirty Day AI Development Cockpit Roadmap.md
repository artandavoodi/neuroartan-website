# Thirty Day AI Development Cockpit Roadmap

---

## 1. Roadmap Purpose

This roadmap defines the thirty-day execution path for developing the AI Development Cockpit inside the Neuroartan website layer.

The roadmap exists to preserve development continuity, reduce dependence on a single external premium tool, and create a modular web-based command surface for prompt composition, terminal scan generation, file targeting, patch planning, model routing, and GitHub workflow support.

---

## 2. Strategic Direction

The website layer is the primary execution surface for this cycle.

The software layer remains strategically important, but it is not the critical path for the current thirty-day continuity mission.

The active operating doctrine is:

- focus on the web app as the development cockpit
- keep software/runtime work as a fallback track
- use Codex through controlled daily prompts
- use ChatGPT as the strategic continuity and surgical correction layer while available
- keep every implementation modular and future-proof

---

## 3. Thirty-Day Mission Outcome

By the end of the thirty-day cycle, the website layer should include a working AI Development Cockpit with the following operational components:

- prompt composer
- terminal scan generator
- file path target selector
- patch instruction generator
- GitHub commit message generator
- open-this-file-next workflow
- model provider abstraction
- cloud provider route
- local provider route
- manual provider route
- Codex daily prompt queue
- daily execution log

---

## 4. Execution Allocation

| Track | Allocation | Execution Role |
|---|---:|---|
| Web App / Website Platform | 70% | Build the cockpit interface, profile/model surfaces, modular UI, and workflow components. |
| API / GitHub / Terminal Bridge | 20% | Build practical repository-aware development workflows and prompt-to-terminal support. |
| Local Runtime / GGUF Fallback | 10% | Preserve local model experimentation as a sovereignty fallback without making it the critical path. |

---

## 5. Week One Focus

Week One establishes the structural foundation.

Primary objectives:

- create the AI Development Cockpit folder/file architecture
- define modular fragments, CSS, JS, and data ownership
- create the prompt composer shell
- create terminal scan generator shell
- create file target selector shell
- create initial provider abstraction specification
- create first Codex prompt for structural implementation

Week One must avoid overloaded files and must not attempt full model integration before the cockpit structure is stable.

---

## 6. Week Two Focus

Week Two turns the cockpit into a usable development workflow surface.

Primary objectives:

- connect prompt composer to reusable prompt templates
- define scan prompt templates
- define patch instruction templates
- define open-file-next workflow
- define GitHub commit message generator
- create execution output panels
- begin cloud provider interface planning

Week Two should prioritize practical development continuity over visual perfection.

---

## 7. Week Three Focus

Week Three introduces provider abstraction and workflow routing.

Primary objectives:

- define provider registry structure
- add cloud provider route
- add local provider route
- add manual provider route
- define provider capability metadata
- create model selection interface
- connect prompt composer to provider mode selection
- document limitations and safe usage boundaries

Provider routing must remain modular and replaceable.

---

## 8. Week Four Focus

Week Four stabilizes the cockpit as a daily development operating system.

Primary objectives:

- refine daily Codex prompt workflow
- connect roadmap to daily execution log
- finalize repository scan templates
- finalize patch instruction templates
- finalize commit message workflow
- verify modular architecture
- document remaining limitations
- prepare next-cycle continuation plan

The end of Week Four should produce a working development cockpit baseline, not a fully autonomous coding system.

---

## 9. Daily Execution Rhythm

Daily execution should follow this sequence:

1. review current roadmap objective
2. open Codex daily prompt queue
3. select one prompt only
4. run Codex against the defined scope
5. test or scan the result
6. return to ChatGPT for surgical review and correction
7. document the daily outcome
8. define the next day’s prompt

No daily session should attempt broad uncontrolled redesign.

---

## 10. Codex Usage Constraints

Codex usage must respect limited availability and rate limits.

Each Codex prompt must be precise, scoped, and structurally bounded.

Every Codex prompt must include:

- exact target root
- exact folder/file creation boundaries
- required modular separation
- forbidden overloading
- forbidden workaround behavior
- required verification output
- expected completion summary

Codex should be used for one controlled structural objective per day whenever possible.

---

## 11. Modular Architecture Rule

The AI Development Cockpit must not become a single large file.

Required modular ownership:

| Layer | Required Pattern |
|---|---|
| HTML | page shell plus fragments |
| CSS | section-specific files plus global token consumption |
| JavaScript | small modules by workflow responsibility |
| Data | JSON registries for providers, prompts, workflows, and routes |
| Documentation | governed planning records under website planning |

Every new file must have clear ownership and a defined reason to exist.

---

## 12. Required Cockpit Modules

The first implementation cycle should create these modules:

| Module | Purpose |
|---|---|
| Prompt Composer | Build structured development prompts from selected intent, target, and constraints. |
| Terminal Scan Generator | Generate scan commands for website, software, provider, and repository scopes. |
| File Target Selector | Select exact files for open/edit workflows. |
| Patch Instruction Generator | Convert an analysis into precise edit instructions. |
| GitHub Commit Message Generator | Generate concise repository commit messages. |
| Open File Next Workflow | Preserve the established exact-path workflow. |
| Provider Router | Select cloud, local, or manual provider mode. |
| Daily Execution Log | Record what was done, verified, blocked, and next. |

---

## 13. Provider Direction

Provider support should begin as an abstraction, not a hard dependency.

Initial provider modes:

| Mode | Purpose |
|---|---|
| Cloud | Free-tier or low-cost model access for high-velocity reasoning and code support. |
| Local | GGUF or compatible local runtime fallback. |
| Manual | Human-mediated copy/paste workflow when automation is not available. |

No provider-specific implementation should be hardcoded into the cockpit shell.

---

## 14. GitHub and Terminal Direction

The cockpit must support repository-aware development workflows.

Required workflow outputs:

- scan command
- target file path
- patch instruction block
- verification command
- commit message
- next file to open

The cockpit does not need direct GitHub automation in the first cycle. It needs reliable human-operated workflow generation first.

---

## 15. Success Criteria

This roadmap is successful if, by the end of the cycle, Artan can use the website layer to:

- generate structured development prompts
- generate terminal scans
- identify file targets
- prepare patch instructions
- prepare commit messages
- route work through cloud, local, or manual provider mode
- maintain daily execution continuity
- continue development even under reduced premium-tool access

---

## 16. Non-Goals

This roadmap does not attempt to complete:

- a full Codex replacement
- a fully autonomous coding agent
- a complete native software runtime
- a full local model development environment
- unrestricted repository write automation
- public exposure of internal development tooling

Those remain future-phase objectives.

---

This roadmap requires founder review before it becomes the active thirty-day execution directive.

After approval, the Codex Daily Prompt Queue becomes the daily operational control file.
