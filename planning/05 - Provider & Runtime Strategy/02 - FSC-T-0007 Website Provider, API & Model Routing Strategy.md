# FSC-T-0007 Website Provider, API & Model Routing Strategy

---

## I. Purpose

This strategy defines how website-facing FSC-T-0007 surfaces must treat API-provider embedding, provider routing, runtime mode, and model routing state.

The active implementation focus is the website/web app only.

Native ICOS software implementation remains paused unless explicitly reactivated by Founder instruction.

---

## II. Core Rule

Provider and API routing are private operational facts.

Public surfaces may show only approved public capability language.

Owner-facing surfaces may show controlled operational state only when that state comes from canonical backend truth.

Website Products data is public category education and commercial awareness.

It is not provider-routing truth, model-routing truth, entitlement truth, permission truth, or economy activation authority.

---

## III. Backend Routing Ownership

Provider/API routing must be owned by backend persistence before UI exposure.

The website/web app must support provider routing through canonical backend records, not public JSON, product copy, or route-local UI state.

The first provider-routing implementation must remain placeholder-only:

- `default_provider`: not assigned;
- `fallback_provider`: not assigned;
- `routing_policy`: private foundation placeholder;
- `runtime_mode`: pending runtime selection;
- `local_runtime_enabled`: false;
- `cloud_runtime_enabled`: false;
- `routing_privacy_state`: private.

No external provider execution is authorized by this strategy.

---

## IV. Private Foundation Routing States

The website private foundation layer must support these routing-related states:

1. model identity registry reference;
2. private model ID reference;
3. public model identity reference;
4. private model identity reference;
5. provider routing placeholder;
6. runtime mode placeholder;
7. entitlement lookup;
8. permission lookup;
9. source authorization lookup;
10. dashboard routing display state;
11. blocked economy routing state.

The first phase may store and display routing status.

It must not execute routing against provider APIs.

---

## V. Public Website Boundary

The public website must not expose:

- private model ID;
- private serial identity;
- private identity ID;
- provider route;
- API route;
- token authority;
- source authorization evidence;
- entitlement record;
- permission record;
- private monetization record;
- private payout record;
- ranking logic;
- inter-model hiring logic;
- posthumous economy logic.

The public website may expose approved high-level language such as:

- private personal model;
- future provider flexibility;
- governed model identity;
- owner-controlled routing readiness;
- model economy as future category education.

---

## VI. Owner-Facing Dashboard Boundary

The owner-facing dashboard may show controlled state labels only:

- provider route: `notAssigned`;
- runtime mode: `pendingRuntimeSelection`;
- routing policy: `privateFoundationPlaceholder`;
- routing privacy: `private`;
- economy state: `blockedUntilReview`.

It must not show raw provider secrets, API routes, tokens, internal keys, provider credentials, or routing authority details.

---

## VII. Entitlement and Permission Relationship

Provider routing must depend on entitlement and permission state.

No provider route may become active unless:

- model identity exists;
- owner authority exists;
- entitlement state allows the route;
- permission state allows the route;
- source authorization is valid where required;
- legal/governance review permits the route class;
- provider/security review permits the route class.

Current FSC-T-0007 state does not authorize provider execution.

---

## VIII. Blocked Economy Boundary

Provider routing must not be used to activate:

- marketplace behavior;
- monetization behavior;
- payout behavior;
- public ranking;
- autonomous inter-model hiring;
- regulated-domain professional behavior;
- guaranteed-income behavior;
- consciousness/personhood claims;
- posthumous economy behavior.

All such states remain blocked until legal, governance, product, security, finance, tax, and implementation review.

---

## IX. Supabase Implementation Implication

The next backend implementation must verify whether existing Supabase migrations already contain provider/runtime tables or fields.

If no canonical provider-routing owner exists, create a new backend migration for private foundation routing state.

The migration must not duplicate existing profile or model ownership.

The migration must initialize provider routing as placeholder-only.

---

## X. Acceptance Criteria

This routing strategy is correctly implemented only if:

- provider routing is backend-owned;
- public JSON does not own provider routing;
- owner-facing dashboard reads controlled state only;
- provider credentials remain secret and absent from public files;
- provider execution remains disabled;
- entitlement and permission checks are structurally represented;
- economy activation remains blocked;
- Supabase migration validation passes.

---
